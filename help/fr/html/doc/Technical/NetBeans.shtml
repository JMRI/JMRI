Introduction

Apache NetBeans constitue une excellente plateforme pour travailler avec JMRI. Le code JMRI est configuré pour fonctionner avec Apache NetBeans sans pratiquement aucune configuration supplémentaire.

Apache NetBeans est un environnement de développement intégré (IDE) très puissant, qui peut rendre le travail avec le code JMRI assez facile. Mais il faut un peu de temps pour s'y habituer, et la lecture des tutoriels sera utile.

Installation

  1. Téléchargez et installez un Java 8 ou plus récent Java Development Kit (JDK) (Il est recommandé d'utiliser un gestionnaire de paquets si possible pour garantir que les mises à jour de sécurité peuvent être installées automatiquement)

  2. Téléchargez Apache NetBeans depuis netbeans.apache.org (Il est recommandé d'utiliser un gestionnaire de paquets si possible pour s'assurer que les mises à jour de sécurité peuvent être installées automatiquement)

  3. Exécuter l'installateur Apache NetBeans

Linux

Apache NetBeans peut également être installé sous forme de paquetsnap pour Linux.

macOS

Apache NetBeans et le JDK Java 8 peuvent être installés en utilisant Homebrew à partir du terminal:

brew cask install adoptopenjdk8 netbeans

Windows

Note

Dû à un traitement différent des caractères de fin de ligne (LF/CR LF) entre Windows et d'autres systèmes - à savoir Unix et OS X (voir Retour Fin de ligne) - il est de la plus haute importance que tous les utilisateurs de Windows adoptent les paramètres suivants pour leur environnement de développement. Cela est absolument nécessaire pour que Git puisse suivre les changements individuels sur les fichiers que vous commettez.

Si vous ne suivez pas ces instructions, vous risquez de fausser les fins de ligne dans les fichiers que vous commettez, qui feront à leur tour que l'historique de leur révision sera beaucoup plus difficile d'accès. Les fichiers dans cet état ne seront pas acceptés comme les demandes d'extraction.

Dans un premier temps, comme JMRI a adopté LF comme convention pour Fin de Ligne , il est nécessaire de définir le paramètre core.autocrlf à true. (voir "Traitement des fins de lignes" sur GitHub Help)

Using Apache NetBeans with JMRI’s Git Code Repository

GitHub fournit le dépôt de code JMRI’s en utilisant l'outil Git. Pour avoir plus d' informations sur la façon dont JMRI utilise Git, voir l'aide de JMRI sur Avoir le code via Git et Git FAQ.

Une fois configuré, Apache NetBeans fournit un support intégré pour l'utilisation de Git.

Pour plus d'informations sur Git et Apache NetBeans, voir la Page d'aide de NetBeans Git

Vérification du code

Dans Apache NetBeans, dans le menu "Team", sélectionnez Git, puis Clone Dépôt. Entrez l'URL du dépôt https://github.com/JMRI/JMRI.gitdu dépôt de la page JMRI GitHub sur le formulaire Apache NetBeans sous "Repository URL". Nous vous recommandons d'entrer également votre nom d'utilisateur et votre mot de passe GitHub sur le formulaire, afin que nous puissions vous attribuer des contributions plus tard. Cliquez sur "Suivant". Assurez-vous que le "master*" est cochée pour faire apparaître la version principale du code. Cliquez sur "Suivant", et cliquez ensuite sur "Terminer".

Il faudra beaucoup de temps pour obtenir une copie du code (notez la barre de progression en bas à droite), mais ensuite vous êtes prêt à commencer.

Local Commits

Lorsque vous utilisez Git, l'opération "commit" est locale à votre ordinateur. Il n’apporte aucune modification au dépôt principal de l’JMRI.

Vous êtes encouragés à committer souvent, afin que vos changements soient stockés en sécurité loin de votre répertoire de travail. Avoir chaque petite modification enregistré séparément peut être incroyablement utile plus tard si vous devez trouver où un problème a été introduit.

Mise à jour du code à partir de Git

De temps en temps, vous devez mettre à jour votre copie locale du code pour correspondent à la "TÊTE" actuelle du contenu principal du dépôt Git de JMRI. Le processus de base consiste à extraire les copies les plus récentes de JMRI de faire les fusions nécessaires si vous avez résolus des conflits et ensuite de l'ajouter à votre propre dépôt GitHub.

Pour ce faire, utilisez les NetBeans Apache [Team] → [Remote] → [Pull depuis Choix de menu [Upstream]. Dites à Apache NetBeans de se déconnecter du "github" distant que vous avez défini ci-dessus.

Vous pouvez également mettre à jour un seul fichier ou le contenu d'un dossier, en en faisant un clic droit sur un élément dans la fenêtre "Fichiers", puis en sélectionnant [Git] → [Remote] → [Pull from Upstream].

Créer une demande d'extraction

Une "demande de retrait" (RP) est un moyen facile de rassembler tous vos changements, même s'ils couvrent plusieurs dossiers, en quelque chose que vous pouvez ensuite soumettre à l'inclusion dans le dépôt principal de JMRI.

Tout d'abord, vous devez créer un compte GitHub et votre propre dépôt.

  • Obtenez un compte GitHub et connectez-vous

    .

  • Demandez à GitHub un "fork" JMRI’s repository dans un des vôtres. Il y a un bonne page d'aide,En gros, vous allez sur la page du dépôt JMRI et cliquez sur le bouton "fork" en haut à droite.

Cela vous donne votre propre dépôt, avec lequel vous pouvez ensuite travailler librement.

Ensuite, poussez les changements jusqu'à votre dépôt. (N'importe qui peut pousser vers son propre répertoire mais la plupart des gens n’ont pas la permission de modifier directement le dépôt principal JMRI ; c’est pourquoi la demande de retrait est nécessaire) Pour faire ceci:

  • Utilisez le choix de menu Apache NetBeans [Team] ; [Remote] ; [Push … ; ​ ;]. (Si vous essayez d'utiliser le choix "Push to Upstream", il essaiera d'écrire sur le dépôt principal, et échouera.)

  • Cliquez sur Specify Git Repository, et entrez l'URL pour votre propre dépôt, par exemple https://github.com/yourname/JMRI.git

  • Cliquez ensuite sur suivant.

  • Apache NetBeans vérifiera avec le référentiel, et affichera une indication de branches à pousser, par exemple "master → master" Cochez la case appropriée pour pour la branche sur laquelle vous avez travaillé.

  • Cliquez sur Suivant et Terminer. Netbeans fera la poussée réelle vers votre dépôt.

Vous pouvez répéter ceci autant de fois que vous le souhaitez, car vous travaillez sur vos changements. À chaque fois, les changements les plus récents sont poussés vers le haut et sont disponibles.

Enfin, lorsque votre changement est prêt à être soumis, allez sur le site web de GitHub pour remplir et soumettre une demande d'extraction.

Éventuellement, quelqu'un examinera la demande de retrait et décidera de la fusionner. Parfois, ils demandent des modifications supplémentaires. Dans ce cas, vous pouvez mettre à jour la demande de retrait en appuyant simplement sur les nouvelles modifications de votre répertoire où ils seront automatiquement récupérés.

Construction et Exécution de JMRI

Astuce:
Assurez-vous que votre ordinateur est connecté à Internet la première fois que vous construisez, testez ou exécutez JMRI après avoir vérifié une nouvelle copie de JMRI sur GitHub afin de permettre le téléchargement automatique de toute dépendance externe ou outil de construction potentiellement manquant.

Prime

La première fois que vous ouvrirez le projet JMRI dans Apache NetBeans, une icône d'avertissement sera affichée sur le projet. Cet avertissement indique qu'Apache NetBeans a un problème avec le projet. Habituellement, le premier avertissement indique que des artefacts de compilation sont manquants.

Pour résoudre ce problème:

  1. Sélectionner le projet JMRI

  2. Choisissez le menu raccourci › Résoudre les problèmes du projet…​ (le menu raccourci est généralement ouvert en cliquant sur le projet avec le bouton droit de la souris) Afficher l'exemple…​

  3. Cliquez Résoudre…​

    Résoudre

  4. Cliquez sur Fermez une fois que l'icône d'avertissement se transforme en icône OK

    .
Note
 Vous pouvez également voir une icône d'erreur si vous utilisez une version du JDK supérieure à 8 en raison de changements dans la façon dont Java interagit avec macOS dans les versions 9 ou plus récentes du JDK.

Construction

Pour construire JMRI, choisissez Run › Projet de construction (JMRI) ou Run › Projet de nettoyage et de construction (JMRI).

Astuce:
Pour ne construire que le code source Java de JMRI, exécutez la phase du cycle de vie de maven compiler.

Exécuter une application

Pour lancer une application JMRI, choisissez Run › Run Project (JMRI) ou cliquez sur l'icône Run dans la barre d'outils. Vous serez invité à exécuter la classe principale JMRI la première fois que vous exécuterez le projet.

Les applications JMRI sont lancées à partir des classes principales:

DecoderPro

apps.DecoderPro.DecoderPro
PanelPro

apps.PanelPro.PanelPro
SoundPro

apps.SoundPro.SoundPro
Astuce:
Une nouvelle vérification de JMRI devrait être construite proprement.. Sinon, veuillez consulter le groupe de développeurs ou le projet pour obtenir de l'aide.

Déboguer une application

Pour déboguer une application JMRI, choisissez Debug › Debug Project (JMRI) ou cliquez sur l'icône Debug dans la barre d'outils. Vous serez invité à déboguer la classe principale JMRI la première fois que vous exécuterez le projet.

Profiler une application

Pour établir le profil d'une application JMRI, choisissez Profil › Profil Projet (JMRI) ou cliquez sur l'icône Profil dans la barre d'outils. La première fois que vous exécutez le projet, la classe principale JMRI vous demandera d'établir un profil.

Faire un fichier Jar

La plus grande partie du code dans une installation JMRI normale vit dans un fichier appelé jmri.jar. Si vous souhaitez remplacer cette version par votre version mise à jour, vous devez créer un nouveau fichier jmri.jar à partir de votre code modifié.

Pour ce faire, ctrl-clic (ou clic droit) sur le nom "JMRI". dans la fenêtre "Projets". Cela ouvrira un menu contextuel, sur lequel vous trouverez "Make Jar File". Sélectionnez cette option, et un nouveau fichier jmri.jar sera produit dans le répertoire principal du projet.

Autrement, dans la fenêtre "Fichiers", faites un clic droit sur le fichier build.xml, et sélectionnez "Run Target", puis "jar".

Faire un paquet de version pour Windows en utilisant NSIS

Il est possible pour les utilisateurs d'Apache NetBeans de créer un paquet de libération pour sur la plate-forme Windows. Cela nécessite l'installation du logiciel Nullsoft Scriptable Install System (NSIS), qui peut être téléchargé à partir de le site web du projet NSIS sur SourceForge.

Une fois que le NSIS est installé sur l'ordinateur, il est nécessaire de créer un fichier appelé local.properties dans le même répertoire que le fichier build.xml de JMRI. Ce fichier n'est pas spécifiquement enregistré via GitHub, de sorte que vos chemins d'installation des outils spécifiques aux OS ne sont pas enregistrés dans GitHub. Créez le fichier, et ajoutez une définition de variable avec le chemin d'accès correct au répertoire qui comprend l'exécutable NSIS, tel que:

nsis.home=C:/Program Files (x86)/NSIS/

ou

nsis.home=/opt/nsis/nsis-2.46/

Note: Assurez-vous que la trace / est présente.

Avec le paquet NSIS installé sur votre ordinateur, et construisez le .xml correctement modifié, le paquet Windows peut être créé en exécutant le "package-windows" ant cible dans build.xml.

Utilisation des outils XML d'Apache NetBeans

Apache NetBeans dispose de bons outils pour travailler avec des fichiers XML, notamment en les éditant et en les validant. JMRI utilise XInclude pour inclure des fragments XML dans d'autres fichiers XML via des URL vers le site web du JMRI. Les programmes JMRI les convertissent au moment de l'exécution en références de fichiers locaux ; cependant, Apache NetBeans doit être configuré pour le faire si vous voulez éditer des fichiers XML JMRI hors ligne.

Pour configurer Apache NetBeans de manière à permettre l'édition XML hors ligne:

  1. Choisissez Outils › DTDs et schémas XML

  2. Cliquez Ajouter le catalogue…​

    .

  3. Sélectionner le Type de catalogue Résolveur de catalogue OASIS

  4. Cliquez Naviguez…​

    .

  5. Naviguer vers /path-to-your-JMRI-project/xml/catalog.xml

  6. Cliquez OK

    Cliquez

  7. Cliquez OK

    Cliquez

  8. Clic Fermer

    Fermer
Remarque:
l'emplacement du catalogue XML n'est pas spécifique à un projet, donc si vous travaillez sur plusieurs projets JMRI, quel que soit le projet que vous ajoutez, c'est le catalogue qui sera utilisé, et non le projet en cours.

Running SpotBugs

L'outil SpotBugs est un outil d'analyse de code statique qui examine le code source JAVA pour les questions de codage communes.

Pour faire fonctionner l'outil SpotBugs sur le code JMRI, exécuter l'objectif maven spotbugs:check qui est défini dans pom.xml. Dans Apache NetBeans Pour ce faire, il suffit de cliquer sur JMRI dans le volet Projets et puis en cliquant sur "spotbugs check" dans le volet Navigator. Lorsque SpotBugs s'achève, la fenêtre de sortie d'Apache NetBeans affichera les résultats SpotBugs .

Chargement d'une branche spécifique depuis GitHub

Le processus de construction à partir d'une branche (c'est-à-dire celle où quelqu'un a mis une modification) est un peu différente des instructions standard. ( Voici NetBeans 8.2, mais il ne varie pas beaucoup selon les versions)

  • USous "Team" → "Remote" sélectionner "Fetch" (pas Fetch from Upstream)

  • Remplissez le formulaire pour spécifier le dépôt de l’utilisateur correct (Après la première fois, vous pourrez faire une sélection par nom dans la partie supérieure choix)

  • Sur l'écran suivant pour sélectionner une branche, vous pouvez soit les sélectionner toutes (il y a un bouton) ou trouver et sélectionner juste le nom de la branche qui maintient le changement souhaité:

  • Cliquez sur "Terminer"

À ce stade, vous avez transféré mes modifications sur votre machine locale, mais vous ne les utilisez pas encore. Pour que ces fichiers apparaissent en local pour Apache NetBeans à construire:

  • Retournez au menu principal, sélectionnez "Team" → "Checkout" → "Checkout Revision…​". Dans la fenêtre qui s'affiche, cliquez sur "Select":

  • En haut à gauche, trouvez la branche importée que vous souhaitez, qui dans l'exemple est "bobjacobsen/ast-clock-power-bit". Cliquez pour mettre en évidence (show) et cliquez ensuite sur "Select", ou double-cliquez simplement dessus. (Ne vous inquiétez pas sur la case supérieure droite ou inférieure, ils seront automatiquement réglés droite)

  • Retour à la fenêtre "Checkout Revision …​ ;", assurez-vous que la fenêtre "Checkout as Nouvelle branche" n'est pas coché, puis cliquez sur "Commander"

Vous avez maintenant une copie de mes modifications comme source de vérification. Vous devrez faire un "projet de nettoyage et de construction" pour s'assurer que vous obtenez une compilation cohérente, et vous devriez être en mesure de l'exécuter.

Important
 Lorsque vous voulez revenir à la version JMRI standard , à partir du dépôt principal, vous avez besoin de vérifier la branche master. Le processus est similaire et plus simple. Choisissez Team › Checkout  ; › Checkout Revision…​, type master dans la case Checkout Selected Revision, et cliquez sur Checkout
