JMRI:Outils USS CTC

Il s'agit de l'implémentation originale de CTC qui fournit des fonctionnalités qui sont activées par le biais de scripts. Voir aussi la nouvelle lise en œuvre de CTC qui fournit un éditeur d'interface graphique pour configurer les règles CTC et un exécutable pour mettre en œuvre la logique CTC. Il construit également un squelette de Panneau dÉdition CTC avec les leviers/indicateurs d'aiguillage, les leviers/indicateurs de signaux, le code bouton , etc, et un plan de voies minimal.

JMRI contient des outils d'aide à l'automatisation des Panneaux USS CTC . Ils peuvent être utilisés soit pour des panneauxs sur écran, soit pour piloter un TCO physique avec de vrais leviers et voyants.

Outils basés sur des scriptsDepuis JMRI 4.8

Ces outils sont très puissants, mais ne peuvent actuellement être configurés que via un script.
Pour savoir comment procéder, voir :

  • L'exemple de script jython/ctc/TwoColumnMachine.py
  • La Documentation Javadoc sur les différentes composantes
Les outils disponibles sont :
Ligne de code/dt>
Met en œuvre la ligne de code entre la machine CTC et le champ. Peut utiliser une sortie d'aiguillage pour indiquer, par exemple, à un voyant de s'allumer ou à des relais de faire du bruit.
Gare
Correspond à une seule gare sur la ligne de code, généralement une seule colonne sur l'appareil CTC et tous les éléments du domaine qui y sont connectés.
Bouton code/dt>
Un par section, cela définit la connexion au bouton code(une entrée de capteur) et le voyant de code (une sortie d'assistance) sur le panneau.
Section de l'aiguillage/dt>
Commande un seul aiguillage sur le terrain. Vous le configurez avec le nom de cet aiguillage ainsi que des entrées de capteurs pour le levier sur le panneau et des sorties d'aiguillage pour les voyants témoins. Peut éventuellement prendre des objets Verrou pour ajouter un verrouillage type.
Section SignalHead/dt>
Commande les feux de signalisation autour d'un seul OS sur le terrain. Vous le configurez avec des listes de Feux gauche et droite qu'il peut forcer et libérer, ainsi que des entrées de capteurs pour le levier sur le panneau et des sorties d'aiguillage pour les voyants témoins. Peut éventuellement prendre des objets Verrou pour ajouter un verrouillage type.
Section de circuit de piste/dt>
Cela fait fonctionner un témoin lumineux d'occupation des voies (une sortie) sur le panneau à partir d'un capteur d'occupation sur le terrain (une entrée de capteur). En option, peut sonner une cloche lorsqu'il est occupé ou inoccupé.
Section d'appel du responsable/dt>
Ceci pilote un témoin d'appel du responsable (une sortie) dans le champ d'un levier ou interrupteur sur le panneau central (une entrée de capteur).
Verrouillage de l'Occupation/dt>
Lorsque cette option est ajoutée à une section Aiguillage ou ou Feu de Signalisation, elle empêche les changements pendant que la section OS (définie dans le Verrou) est occupée.
Verrouillage d'Itinéraire
Ajouté à une section d'Aiguillages, cela empêche les changements tandis que la section OS a un itinéraire dégagé à travers l'Aiguillage.
Blocage horaire
Ajouté à une section d'Aiguillage, cela empêche les changements tandis que la section OS a des signaux en cours d'exécution.
Verrouillage Trafic/dt>
Se verrouille contre les signaux qui sont libérés dans des directions opposées sur un seul segment de voie. Peut être configuré avec les réglages de l'Aiguillage (objets BeanSetting) pour permettre, par exemple, à des signaux aux extrémités opposées d'une Ville de libérer lorsqu'un aiguillage est dirigé sur la voie principale et l'autre sur la voie d'évitement.
Verrouillage d'Aiguillage
Un terme un peu inapproprié, car il fait référence à un verrouillage basé sur l'état d'un Aiguillage JMRI, qui doit être considéré comme un résultat généralisé. Peut être utilisé, par exemple, pour verrouiller les sections où un état Appel On (sur un Aiguillage interne qui représente l'état) a été défini (par exemple, par une section d'appel du responsable utilisée pour définir cet Aiguillage JMRI sur le terrain).
Il y a une page avec un exemple travaillé (pas encore présent), et une page séparée décrivant les conventions d'appellation utilisées dans l'exemple, au cas où vous constaterez qu'elles vous épargneront un certain travail de mise en place.

Ces outils enregistrent des informations dans plusieurs objets de la mémoire JMRI, qui peuvent à leur tour être placés sur des panneaux en tant que MemoryIcons.

IMUSS CTC:SIGNALHEADSECTION:1:LOG
Ce que fait la ligne de code
IMUSS CTC:SIGNALHEADSECTION:1:TIME
Quand le temps passe
IMUSS CTC:LOCK:1:LOG
Indication de verrouillage des verrous
Le "1" peut changer s'il y a plus d'un ensemble d'objets, c'est-à-dire si plus d'une ligne de code est modélisée.

Outils autonomes

Ceux-ci peuvent être créés à partir du menu USS sous Outils dans Applications JMRI. Ils travaillent en créant des applications Logix et/ou des itinéraires qui mettent en œuvre la fonctionnalité souhaitée. Vous pourriez le faire à la main, bien sûr, mais ils sont conçus comme une simplification.
Follower
Esclave une sortie Dispositif d'Aiguillage pour suivre l'état d'un dispositif de capteur d'entrée. Cela pourrait, par exemple, rendre Les panneaux d'affichage suivent un interrupteur de façade relié à un Sensor.
Indicateur OS
Prend les données d'un capteur pour montrer que la section de la voie est occupé, et peut également, en option, allumer l'indicateur si la section a été déverrouillée.
