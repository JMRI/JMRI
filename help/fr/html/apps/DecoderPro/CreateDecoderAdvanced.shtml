Définitions avancées des décodeurs

Cette page fournit des informations sur les fonctionnalités avancées des fichiers de définition du décodeur pour le Programmeur DecoderPro Symbolic.

Formatage sur la page

Vous pouvez faire un formatage limité sur l'écran en utilisant des balises de formatage HTML.

Vous pouvez intégrer du HTML dans le texte du volet, mais il doit être inclus dans les balises <html>...</html>.

Cela est rendu un peu délicat par le chevauchement entre les "balises faisant partie de la structure XML du fichier décodeur" et les "balises HTML inclus dans le fichier du décodeur". Toutes les balises HTML doivent être cités, avec leurs caractères < et > remplacés par les &lt; et &gt; les chaînes de caractères et tout double-note intégré étant remplacés par &quot;.

Par exemple:

   <label label="&lt;html&gt;&lt;strong&gt;JMRI 3.7.2 ou une version ultérieure est nécessaire pour utiliser cette définition.&lt;/strong&gt;&lt;/html&gt;"/>
Après la lecture du XML, le contenu de cette étiquette sera affiché à partir du texte de l'attribut "label", qui est : 
     <html><strong>Pour utiliser cette définition, il est nécessaire de disposer de JMRI 3.7.2 ou d'une version plus récente.</strong></html>
et apparaîtra à l'écran comme

JMRI 3.7.2 ou supérieur est nécessaire pour utiliser cette definition.

Pour plus d'exemples, voir xml/decoders/esu/v4readMePane.xml.

Il ne prend en charge que le HTML5, donc si quelque chose ne s'affiche comme vous l'attendez, voyez si une approche utilisant du HTML plus simple fonctionnera.

Identification

Lorsque la fonction "Lire le type de décodeur" est utilisée pour déterminer le fichier de définition du décodeur, plusieurs informations sont lues à partir du décodeur et utilisées pour tenter une correspondance avec une famille et un modèle :
Fabricant:
Valeur numérique de CV8, qui est ensuite mise en correspondance avec un nom en utilisant les entrées du fichier xml/decoderIndex.xml file.
Code de version:
Valeur numérique de CV7, qui est ensuite mise en correspondance avec le(s) nom(s) de modèle(s) utilisant les ID de version basse et haute sur les éléments du modèle dans toutes les définitions de décodeur pour le fabricant spécifique. Il s'agit d'un processus complexe qui ne se passe pas toujours bien en raison de valeurs doubles, etc.
Code d'identification du produit:
A l'origine, une valeur numérique lue à partir d'un ou des CV spécifiques au fabricant; cela ne fonctionne que pour certains fabricants. Le processus de lecture est traité dans le code (le jmri.jmrit.decoderdefn.IdentifyDecoder classe) et à mettre à jour chaque fois qu'un fabricant commence à fournir dans un ou plusieurs nouveaux CV. Lorsque le code produit est présent, il devrait briser toute dégénérescence entre le chevauchement des valeurs CV7 au sein d'une famille de décodeurs. Par exemple : 
  <decoder>
    <family name="My Decoder Family" mfg="NMRA"> ;
      <model model="Model A" lowVersionID="16" highVersionID="23" productID="400" />
      <;/model>
      <model model="Model B" lowVersionID="21" highVersionID="27" productID="401" />
      color=;/model>;model model="401" />;model model="401" />
    </family>
    … ;
  </decoder>

Certains décodeurs peuvent avoir plusieurs ID de produits stockés possibles pour des décodeurs identiques. Dans ce cas, plusieurs décodeurs séparés par des virgules peuvent être spécifiées dans le champ "Product ID" et une correspondance avec une valeur quelconque de la liste identifiera le décodeur de manière unique. Par exemple, la série ESUseries utilise les CV RailCom® ; Product ID, mais le Product ID est mis à ; jour pour différents lots de production du même modèle:

  <decoder>
    <family name="ESU LokSound V4.0" mfg="Electronic Solutions Ulm GmbH" lowVersionID="255" highVersionID="255"} ;
      <model model="LokSound V4.0" numOuts="92" numFns="40" extFnsESU="yes" productID="33554493,33554503,33554538"} ;
      </model>
      <model model="LokSound V4.0 M4" numOuts="92" numFns="40" extFnsESU="yes" productID="33554500,33554536"} ;
      </model>
    </family>
    … ;
  </decoder>

WAvertissement: Ne pas utiliser plusieurs Product IDs stockés possibles pour des décodeurs similaires qui nécessitent une expansion conditionnelle différente. Cela ne fonctionnera pas, pour les raisons expliquées dans une section ci-dessous. Dans ce cas, chaque ID de produit doit se voir attribuer un nom de modèle différent, comme dans le premier exemple ci-dessus.

Le support est également ajouté pour trouver cette valeur en utilisant le protocole LocoNet SV2 un certain temps pendant la diffusion de la version 4.1 de JMRI série. Voir les notes de publication pour plus d'informations sur la série progrès.

Puisque le code du produit fonctionne avec l'inclusion/exclusion décrites ci-dessous, vous pouvez également l'utiliser dans les cas suivants où il n'y a pas de CV ou de CV qui le contiennent. Dans ce cas, aucune identification automatique n'est effectuée, mais la valeur du productID du modèle sélectionné par l'utilisateur est utilisé lors de la le traitement et l'extension de la définition du décodeur. Des valeurs alphanumériques productID sont parfois utilisées à cette fin afin d'éviter les conflits avec les chiffres de CVs.

Quand un modèle de décodeur est sélectionné, soit manuellement, soit à la suite du processus d'identification, les noms de famille et de modèle sont stockés dans l'entrée de la liste en cours de création et sont utilisés pour relier à l'avenir une entrée de la liste à une définition du décodeur.

Il est important de noter que le code du fabricant, de la version ou Les ID de produits ne sont pas enregistrés dans l'entrée du fichier, car les valeurs ne sont pas déterminé de manière fiable dans certains cas (par exemple, sélection manuelle, mise à jour du micrologiciel par un logiciel fourni par le fabricant, informations d'identification mises à jour par le vendeur). Cela a des implications pour le moment où la définition du décodeur est élargie pour la programmation (Définitions des décodeurs conditionnels) ou mis à jour (Mises à jour et migration).

Décodeurs du domaine public et du bricolage

Depuis JMRI 4.11.5

La fonction "Lire le type de décodeur" est disponible pour les décodeurs du domaine public et les décodeurs de bricolage.

Trois identificateurs sont utilisés pour identifier le décodeur:

Fabricant:
Valeur numérique CV8 = 13 selon la norme NMRA S-9.2.2. fichier.
Code de version:
La valeur numérique de CV7 où la version du logiciel est stockée.
Code d'identification du produit:
Valeur numérique stockée dans CV47, CV48, CV49 et CV50 selon le fichier de la norme S-9.2.2 de l'ARNM et réservé à l'usage du fabricant.

L'ID du produit est calculé à l'aide de la formule suivante:

  • ID du produit = (((CV47 * 256) + CV48) * 256 + CV49) * 256 + CV50

Les CV individuels ont la signification suivante:

  • CV47 - code pays, par exemple CV47 = 1 pour la République tchèque
  • CV48 - code du développeur ou du groupe de développeurs, tel que CV48 = 1 pour dccdoma.cz
  • CV49 - troisième ProductID pour identification ultérieure
  • CV50 - quatrième ProductID pour une identification plus poussée

Le fichier xml de définition du décodeur correspondant a un élément rempli de type <family>. Exemple:

  <decoder>
    <family name="dccdoma.cz" mfg="Public-domain and DIY" >
      <model model="UNI16ARD-NAV" lowVersionID="41" productID="16843010" … ; />
    </family>
    … ;
  </decoder>

Les attributs individuels de cet élément doivent être remplis comme suit:

  • mfg="Public-domain and DIY" qui correspond à CV8
  • lowVersionID="nn" highVersionID="mm" qui correspond à CV7
  • productID="nnnnnn" qui correspond aux CV47, CV48, CV49 et CV50
    • .

Le nom du fichier xml avec la définition du décodeur doit avoir la structure suivante:

Public_Domain_family-name_model-model.xml

où chaque partie d'un nom de fichier a la signification suivante:

  • Public_Domain qui correspond à l'attribut <family mfg="Public-domain and DIY" >
  • family-name qui correspond à l'attribut <family name="developer" >
  • model-model qui correspond à l'attribut <model model="decoder name" >

Exemple de fichier de décodage de définition pour le décodeur de signal UNI16ARD-NAV.xml du groupe de développeurs dccdoma.cz qui sont originaires de la République tchèque Public_Domain_dccdoma_UNI16ARD_NAV.xml.

Pour le nouveau code pays et le nouveau code développeur, veuillez contacter Groups.io jmriusers https://groups.io/g/jmriusers.

Considérations sur les volets

Les volets du programmateur sont utilisés pour diviser et organiser les variables du décodeur en noms de catégories pratiques .
  • JMRI fournit un ensemble standard de volets, défini par <pane> inclus dans l'élément <programmer> de chacun des fichiers standard de Programmer (Basic, Comprehensive, Advanced, etc.) dans le fichier xml/programmeurs/ .
  • Les fichiers de définition du décodeur peuvent contenir des volets personnalisés pour des décodeurs particuliers ou des familles de décodeurs, définis par <pane> éléments inclus dans le <decoder-config> élément du fichier décodeur ( haut niveau ) file.
  • Les noms des volets sont spécifiés par un élément de type <name> en haut de chaque élément <pane>.

Remplacement des volets

Des volets personnalisées sont généralement ajoutées à pour l'une des raisons suivantes :
  1. Les éléments à montrer ne font partie d'aucun des les catégories définies par les noms standard des volets.
    • Dans ce cas, le <name>de l'élément volet doit être différent de tout nom standard existant.
    • Le nouveau volet personnalisé sera ajouté à la fin de la pane liste des volets.
  2. La catégorie de panneau standard applicable n'a pas la une mise en page correcte ou un nombre de variables correct.
    • Dans ce cas, le <name>de l'élément volet doit correspondent exactement à la norme existante de <e;name> élément volet.
    • Le nouveau volet personnalisé remplacera l'actuel volet standard du même nom.

Volets vides

Les volets vides sont définis comme ceuxs qui n'ont pas d'éléments variables actifs à afficher pour le décodeur sélectionné. Ils peuvent encore contenir du contenu textuel.

Visibilité des volets

La visibilité des volets est régie par un un ensemble de règles complexes
  • Indépendamment de toutes les autres règles, un volet ne montrera pas en cas d'échec "inclure" ou "exclure" les attributs dans son élément <pane> .
  • La visibilité du volet vide est contrôlée par les éléments suivants, en ordre de priorité croissant :
    • Les Préférences->Liste->Programmeur->Afficher Paramétrage des onglets vides.
    • Si présent, un attribut showEmptyPanes dans le fichier XML du programmeur actuel.
    • Si présent, un attribut showEmptyPanes dans le fichier XML de définition spécifique du décodeur .
  • S'il s'agit d'un volet vide spécifique (tel que défini dans Panneaux Vides ci-dessus) il dépend également du type de panneau :
    • Si le volet vide est un volet standard (tel que défini dans Considérations sur les panneaux ci-dessus), il n'apparaîtra que si showEmptyPanes est dans force. Cependant, son contenu habituel sera remplacé par un message générique expliquant que le volet est inutilisé puisque le contenu est inapplicable ou figure sur un autre volet.
    • Si le volet vide est un volet de remplacement (tel que défini dans le point B de Volet Remplacement ci-dessus), il n'apparaîtra que si showEmptyPanes est en vigueur.
    • Si le volet vide est un volet personnalisé (tel que défini dans item A of Volet Remplacement ci-dessus), il apparaîtra toujours, indépendamment du fait que showEmptyPanes soit en vigueur.

Bonne Pratique

Les utilisateurs avancés peuvent choisir de cacher le vide mais l'option Afficher les onglets vides est activée par défaut, donc les utilisateurs débutants doivent savoir pourquoi les volets sont vides ou où trouver les éléments qui ne figurent pas dans les volets prévus.

La meilleure pratique recommandée est:

  • Dans la mesure du possible, les articles doivent utiliser la norme existante des volets .
  • Lorsqu'un volet standard existant particulier ne convient pas en termes de nombre d'articles ou de mise en page :
    • Créer un panneau de remplacement avec le même nom que le panneau standard existant.
    • Pour des exemples, voir les volets "Avancé" et "Son" de la définition du décodeur ESU_LokSoundV4_0 .
  • Lorsqu'un volet standard existant ne convient pas en termes de concept :
    • Créer un ou plusieurs nouveaux panneaux personnalisés avec des noms descriptifs.
    • Créer des informations uniquement (pas d'affichage de variables éléments, juste du texte) volets de remplacement avec les mêmes noms que les volets standards existant non utilisés.Utilisez ces panneaux pour expliquer pourquoi chaque volet est inutilisé et/ou où trouver les informations qu'un utilisateur peut raisonnablement espérer trouver sur ce volet. Utilisez le formatage HTML pour améliorer l'affichage de ces volets explicatifs.
    • Pour des exemples, voir les volets "Lumières" et "Niveaux sonores de la definition du décodeur ESU_LokSoundV4_0.
  • Information seulement Volets personnalisés will always be displayed.

Dans certains cas, par exemple un décodeur non-locomotive pour les servos où aucune des volets du décodeur de locomotive n'a de sens, de sorte que la définition du décodeur contient beaucoup de volets personnalisés, c'est une simplification pour mettre un attribut showEmptyPanes="no" dans l'élément de la définition XML <decoder-config> et juste cacher tous ces volets vides par défaut.

Insertion des fragments de fichier f

Parties communes du décodeur Les définitions peuvent être placées dans des "fragments de fichiers ", qui peuvent ensuite être inclus dans de multiples fichiers de définition de décodeurs. Ce qui facilite la maintenance et la mise à jour de ces définitions, par exemple pour les traduire dans d'autres langues.

Par convention, nous mettons ces fichiers de fragments dans sous-répertoires:

xml/décodeurs/nmra
Les définitions NMRA pour les CV, utilisées par de nombreux décodeurs types
xml/décodeurs/(mfg)
Variables spécifiques aux fabricants, par exemple xml/decoders/esu
Le format d'une inclusion est : 
     <xi:include href="http://jmri.org/xml/decoders/nmra/shortAndLongAddress.xml"/>
Le programme recherche d'abord ces fichiers dans le répertoire des préférences, puis dans le répertoire des programmes doit être présent, par exemple vous avez besoin d'un répertoire xml qui contient un répertoire des décodeurs dans votre répertoire de préférences pour le fragment à trouver.

Utiliser un fichier existant comme exemple pour les éléments supérieurs & format nécessaire.

Mises à jour et migration

Parfois, nous remplaçons une définition dedécodeur par une meilleure. L'ancienne est en usage, mais nous avons décidé de faire passer ces utilisateurs à la nouvelle. Il existe des attributs de niveau-modèle qui peuvent être ajoutés à l'ancienne (et non à l nouvelle) définition pour faciliter cela : 
    <model show="no" model="A4X" replacementModel="A4X" replacementFamily="Jan 2012"> ;
Lorsque l'utilisateur sélectionne "Mettre à jour les définitions du décodeur" dans le menu Debogage dans la fenêtre PanelPro ou le menu Action dans DecoderPro, tous leurs entrées existantes dans l'inventaire qui utilisent une définition de décodeur avec replacementModel et/ou replacementFamily définis seront mis à jour seront mis à jour et remplacés par la définition du décodeur de remplacement spécifié. Cela permet aux gens de continuent à utiliser leur ancienne et confortable définition de décodeur l'ancien dossier reste dans JMRI) jusqu'à ce qu'ils décident de passer à un nouveau.

Notez qu'une mise à jour définie de cette manière est inconditionnelle ; si les attributs de la famille et du modèle correspondent à une entrée de l'inventaire, elle sera mise à jour en fonction de la nouvelle famille/modèle. Les mises à jour conditionnelles automatisées basées sur la versionID ou la productID ne sont pas possibles car ces valeurs sont non stockées dans l'entrée de l'inventaire. Un processus de mise à jour interactif est désormais disponible dans ces cas et utilise une syntaxe spéciale : Depuis JMRI 4.9.5 

    <model show="no" model="DP2X" numOuts="2" numFns="2" replacementFamily="query:Jan 2000|June 2007|Jan 2012"} ;

Un champ de remplacement commençant par la chaîne "query:", éventuellement (de préférence) suivi d'une barre verticale "|" liste séparée des remplacements possibles, déclenche des dialogues interactifs avec l'utilisateur, contenant des explications et des conseils supplémentaires pour l'utilisateur. Cliquez sur ici et ici pour des exemples typiques de ces dialogues. Les définitions manquantes des décodeurs seront également détectées. Cliquez sur ici et ici pour plus d'exemples.

L'attribut "show" de l'élément de modèle permet de s'assurer que les nouveaux utilisateurs commencent avec la nouvelle définition. Ses valeurs possibles sont:

  • "oui" (par défaut) - le comportement traditionnel, toujours montrer ce modèle et la famille qu'il contient
  • "non" - ne jamais montrer ce modèle, même s'il correspond à une sélection possible. Utilisé pour les (anciennes) définitions qui ont été remplacées par de nouvelles.
  • "peut-être" - à n'afficher que si elle correspond à une (possible) identification automatique.
Créer une définition entièrement nouvelle pour un modèle :
  1. Décidez où vous voulez que le nouveau modèle soit défini. Il doit avoir soit un nouveau nom de modèle, soit un nouveau nom de famille (nouveau dossier) ou les deux.
  2. Créez votre définition.
  3. Ajouter "replacementModel" et "replacementFamily attribue à l'ancienne définition.
  4. Généralement, placer l'attribut "show" dans l'ancien définition à "non". La seule exception à cette règle est lorsque certains utilisateurs voudront sélectionner l'ancienne définition encore pour un nouveau décodeur, peut-être parce que l'identification automatique peut distinguer les utilisations de l'ancienne définition de la nouvelle. (par exemple vous divisez l'ancienne définition en deux parties parce qu'une nouvelle version de la locomotive existe)

Capacités de programmation

Certains décodeurs peuvent faire plus d'opérations de programmation, par exemple en contournant les limitations de Programmation des CV dans la Centrale. Les éléments "Capability" sont utilisés pour en informer DecoderPro.

Accès aux adresses CV élevées

Certaines Centrales ne peueuvent accéder qu'au CV 256 ou CV512. Les fabricants de certains décodeurs ont intégré des mécanismes permettant de contourner ce problème. Par exemple, certains décodeurs ESU peuvent utiliser des écritures sur divers CV pour accéder à des CV à haute adresse. Pour indiquer à DecoderPro qu'un décodeur peut le faire, utiliser un élément comme : 
    <capability>
        <name>High Access via Double Index</name>
        <parameter name="Max Regular CV address">256</parameter>
        <parameter name="High Address CV">96</parameter>
        <parameter name="Low Address CV">97</parameter>
        <parameter name="Access Register CV">99</parameter>
        <parameter name="Modulo">100</parameter>
    </capability>

Les paramètres doivent apparaître dans l'ordre indiqué. Ils mean:

  • "Adresse CV ordinaire maximale" - adresses CV égales ou inférieures ce n'est pas ce mécanisme qui est utilisé. (Le mécanisme ne sera pas utilisé si le poste de commandement peut directement lire/écrire le CV étant mais même si ce n'est pas le cas, ce mécanisme ne sera pas utilisé en dessous de cette valeur)
  • "High Address CV" - La partie haute (voir Modulo) du L'adresse du CV est inscrite ici au début de la opération
  • "CV à faible adresse" - La partie basse (voir Modulo) du CV L'adresse est écrite ici comme la 2ème partie de l operation.
  • "Access Register CV" - après les valeurs d'adresse sont set, c'est le CV qui est lu ou écrit pour lire ou set le CV réel souhaité.
  • "Modulo" - valeur utilisée pour diviser l'adresse réelle du CV entre les CV à haute et basse adresse.

Par exemple, pour écrire 21 à CV 614 avec les valeurs indiquées, DecoderPro va:

  • Ecrire 6 =(614/100) au CV 96 pour l'adresse haute part
  • Ecriture 14 = (614 mod 100) au CV 97 pour l'adresse basse part
  • Ecrivez 21 à CV 99 pour faire le CV réel write

Pour un exemple de positionnement de cette capacité dans le fichier, etc., voir l'élément xml/decoders/ESU_LokSoundV4_0.xml fichier.

Autre L'approche utilisée par Zimo, par exemple, consiste à fournir un moyen de mettre en page le l'accès. Par exemple, pour écrire 21 à CV614 avec l'un d'entre eux, vous devez:

  • Ecrire 60 =(614/100)*10 au CV 7 pour l'adresse élevée part
  • Ecrivez 21 dans le CV 14 (614 mod 100) pour faire le CV réel write
Cet élément de capacité fera cela : 
        <capability>
            <name>High Access via Partial Index</name>
            <parameter name="Max Regular CV address">100</parameter>
            <parameter name="High Address CV">7</parameter>
            <parameter name="High Address Multiplier">10</parameter>
            <parameter name="Modulo">100</parameter>
        </capacité>
Note qu'il a une valeur de "nom" différente. C'est ce que sélectionne les différents algorithmes nécessaires ici.

Les paramètres doivent apparaître dans l'ordre indiqué. Ils signifient:

  • "Max Regular CV address" - Les adresses de CV égales ou i nférieures n'utilisent pas son mécanisme (Le mécanisme ne sera pas utilisé si la centrale peut directement lire/écrire le CV CV demandé, mais même s'il ne le peut pas, ce mécanisme ne sera pas utilisé en dessous de cette valeur)
  • "High Address CV" - La partie haute (voir Modulo) de l'adresse du CV est inscrite ici au début de l'opération
  • "Low Address CV" - La partie basse (voir Modulo) de l'adresse du CV est écrite ici comme la 2ème partie de l'opération.
  • "Access Register CV" - après que les valeurs d'adresse aient été définies, c'est le CV qui est lu ou écrit pour lire ou définir le CV réel souhaité.
  • "Modulo" - valeur utilisée pour diviser l'adresse réelle du CV entre les CV à adresse haute et basse.

Par exemple, pour écrire 21 dans CV 614 avec les valeurs indiquées, DecoderPro va:

  • Ecrire 6 =(614/100) au CV 96 pour la partie haute de l'adresse
  • Ecrire 14 = (614 mod 100) au CV 97 pour la partie basse de l'adresse
  • Ecrire 21 dans CV 99 pour faire écrire le CV réel

Pour un exemple de l'endroit où placer cet élément de capacité dans le fichier, etc., voir le xml/decoders/ESU_LokSoundV4_0.xml fichier.

Une autre approche, utilisée par exemple par Zimo, consiste à fournir un moyen de paginer l'accès. Par exemple, pour écrire 21 à CV614 avec l'un d'entre eux,, vous devez:

  • Ecrire 260 =(614/100)*10+200 à CV 7 pour la partie haute de l'adresse. "200" est un indicateur pour utiliser ce mode.

  • "High Address CV" - La partie haute (voir Modulo) de l'adresse du CV est inscrite ici au début de l'opération
  • "High Address Multiplier" - Ceci multiplie la partie haute de l'adresse du CV avant qu'elle ne soit stockée dans le CV de l'adresse principale. Par exemple, en mettant un 10 ici, on peut stocker la partie "haute adresse" dans le "10 ième digit" de l'Adresse Haute CV.
  • "Modulo" - valeur utilisée pour diviser l'adresse réelle du CV entre les CV à adresse haute et basse .
  • "Indicator" - valeur ajoutée à la valeur stockée dans l'adresse haute du CV pour indiquer au décodeur d'utiliser ce mode.

Accessory Decoder Ops Mode

Certains décodeurs accessoires nécessitent des paquets de programmation spéciaux pour configurer leurs CV à partir du signal de la voie DCC. La capacité de faire cela est écrite : 
        <capability>
            <name>Ops Mode Accessory Programming</name>
            <parameter name="Address Type">decoder</parameter>
            <nom du paramètre="Delay">200</paramètre>
        </capacité>
Le paramètre "Address Type" est facultatif :
  • Si elle est "décoder" ou omise, l'adresse actuelle est interprétée comme une adresse de décodeur accessoire de 9 bits.
  • Si elle est "accessory" ou "output", l'adresse actuelle est interprétée comme une adresse de sortie accessoire de 11 bits.
  • Si c'est un "signal", l'adresse actuelle est interprétée comme une adresse de décodeur de signal de 11 bits.
  • Si elle est "altsignal", l'adresse actuelle est interprétée comme une adresse de décodeur de signal de 11 bits, en utilisant une autre interprétation de NMRA standard 9.2.1).
  • S'il s'agit d'un "héritage", l'adresse actuelle est interprétée comme une adresse sur 9 bits (comme pour le "décodeur") mais un paquet de programmation "héritée" (selon l'annexe A de NMRA standard 9.2.1) est envoyé.
Le paramètre "Retard" est facultatif et spécifie un retard minimum (en millisecondes) entre des écritures successives. Certains décodeurs manqueront des commandes si le délai est trop court. Si ce paramètre est omis, le délai par défaut est de 500 millisecondes.

Notez que cela ne fonctionne que pour les écritures en mode ops; ces paquets DCC NMRA particuliers ne fonctionnent ni en mode lecture, ni en mode service (voie de programmation).

Notez un élément : 
        <variable … ; item="Short Address" …> ;
            … ;
        </variable>
ou : 
        <variable … ; item="Long Address" …> ;
            … ;
        </variable>
est nécessaire pour que le programmeur en Mode Ops ait une adresse à utiliser. Les deux formulaires fonctionnent de manière identique. Le paramètre "Address Type" est le seul moyen de spécifier si l'adresse est sur 9 ou 11 bits.

Pour un exemple de travail, voir le xml/decoders/ESU_SwitchPilot.xml fichier. Dans le cas de cet exemple de décodeur, l'adresse est stockée dans les CV 1 et 9 selon Norme NMRA 9.2.2.

Programmation retardée en Mode Ops

La plupart des décodeurs de locomotives acceptent des paquets de programmation spéciaux pour configurer leurs CV à partir du signal de voie DCC (Programmation sur la voie principale).

Toutefois, ces paquets prennent du temps à traiter et si des paquets ultérieurs sont envoyés trop tôt, on peut les manquer. La possibilité d'ajouter un délai après la programmation de chaque CV est écrite : 
        <capability>
            <name>Ops Mode Delayed Programming</name>
            <parameter name="Delay">500</parameter>
        </capacité>
Le paramètre "Delay" spécifie un délai minimum (en millisecondes) entre les écritures successives. Certains décodeurs manquent des commandes si le délai est trop court. Si ce paramètre est omis, le délai par défaut est de 500 millisecondes.

Notez que cela ne fonctionne que pour les écritures en mode ops; ces paquets DCC NMRA particuliers ne fonctionnent ni en mode lecture, ni en mode service (voie de programmation)..

Il convient également de noter que cette capacité particulière ne fonctionne que pour les décodeurs multifonctions. Les décodeurs accessoires doivent utiliser le Mode d'exploitation du décodeur d'accessoires qui possède son propre paramètre "délai".

Accès au CV indexé

Pour accéder aux CV indexés, il suffit d'inclure cet élément de capacité : 
        <capability>
            <name>Indexed CV access</name>
            <parameter name="PI">31</parameter>
            <parameter name="SI">32</parameter>
            <parameter name="cvFirst">false</parameter>
            <parameter name="skipDupIndexWrite">true</parameter>
        </capacité>

Si cvFirst est vrai, le format est CV.PI ou CV.PI.SI tel qu'utilisé par QSI. Si elle est fausse, le format est PI.CV ou PI.SI.CV tel qu'utilisé par ESU.

Les numéros de CV PI et/ou SI peuvent également être définis en utilisant une syntaxe "nn=nn" lors de la spécification PI et/ou SI. Par exemple, en utilisant une fausse syntaxe cvFirst, "101=12.80" met CV101 à 12 avant l'accès au CV 80, quelle que soit la valeur PI configurée dans la capacité.

Si skipDupIndexWrite est vrai, certaines écritures dans le PI et le SI les CV peuvent être ignorés si l'utilisateur choisit de le faire. Cela permet d'accélérer l'accès. Certains décodeurs exigent que les CV soient écrits pour chaque opération, dans ce cas, il faut le mettre sur false. La valeur par défaut est true.

Si ce paramètre et les capacités d'"accès aux CV élevés" sont présent, celui-ci doit être listé en deuxième position.

Pour un exemple de positionnement, etc. xml/decoders/ESU_LokSoundV4_0.xml fichier.

Accès au décodeur TCS WOW

Les décodeurs TCS WOW utilisent plusieurs méthodes uniques (pour le moins) d'accès au CV. JMRI les accueille par le biais d'un élément de capacité spéciale : 
        <capability>
            <name>TCS 4 CV access</name>
        </capability>
Cela fournit deux formats de CV personnalisés :
  • T2CV.11.12 écrit 11 au premier index CV (201), 12 au 2ème index CV (202), puis fait l'écriture/la lecture/confirmation sur les CV 203 et 204
  • T3CV.11.12.13 écrit 11 au premier index CV (201), les données au 2ème index CV (202), puis écrit 12 au CV203 et 13 à CV204.
Pour des exemples d'utilisation, voir voir la définition du décodeur WOW du TCS..

splitVal exemple - adresses "longues" sur 2 CVs

Certains décodeurs, principalement des décodeurs accessoires, utilisent des adresses "longues" réparties sur 2 CV, contigus ou non..

CV1 et CV9 sont normalisés par le NMRA pour l'adresse primaire des décodeurs accessoires, voir la Norme NMRA 9.2.2.

Les adresses longues sont accessibles via le type "splitVal", comme dans l'exemple suivant:

    <variable CV="1" item="Short Address" default="1" minOut="1">
        <splitVal highCV="9" upperMask="XXXXXVVV"/>
        <label>Accessory Address</label>
    </variable>

Le "CV" déclaré est le LSB (octet le moins significatif) alors que le "highCV" est le MSB (Most Significant Byte).

splitVal a en fait deux paramètres, un "facteur" et une " Offsetse ":

  • Valeur à mettre dans les CV = ((valeur dans le champ de texte) - Compensation)/Facteur
  • Valeur à mettre dans le champ de texte = ((valeur dans les CV) * Facteur) + Offset
    <splitVal highCV="9" factor="4" offset="-3"/>

Pour un exemple, voir le fichier xml/decoders/SPROG_DCC_Servo6.xml f.

Définitions des décodeurs conditionnels

Pour réduire le nombre des fichiers de définition de décodeur nécessaires, chaque fichier peut contenir les définitions de plusieurs modèles de décodeurs. JMRI fournit des moyens d'adapter la définition à des modèles spécifiques, tant au moment où le fichier est initialement développé pour être utilisé, que plus tard pendant que le programme travaille sur la définition. Ces mesures sont particulièrement utiles lorsqu'on utilise un fragment de fichier commun qui pourrait être inclus par de multiples fichiers de définition de décodeur (définissant des familles différentes mais similaires)

Inclure/Exclure pendant l'extension de la variable

L'extension et l'utilisation de l'élément variable peuvent être conditionnées par le type de décodeur en utilisant les attributs "inclure" et "exclure sur les éléments ; variables> ,,variable>, sur <enumChoice> et sur <enumChoiceGroup> .

Une définition de variable ne sera pas étendue et utilisée:

  • Si un attribut d'exclusion est présent, ni l'ID produit du décodeur, ni le nom du modèle, ni nom de famille n'est trouvé dans la liste d'exclusion.
  • Si un attribut include est présent, au moins un des numéro d'identification du produit, le nom du modèle ou le nom de famille du décodeur est trouvé dans la liste d'inclusion.
Ces deux conditions doivent être remplies, et il est possible d'utiliser à la fois l'inclusion et l'exclusion sur le même élément, mais il est habituel de n'en utiliser qu'un seul.

Exemple:

      <variable label="My Variable" CV="72" include="ModelA,400" ;
        <decVal/>
      </variable>

La variable ci-dessus n'apparaîtra que lors de l'expansion de la définition du décodeur pour le modèle A ou l'ID de produit 400.

Vous pouvez également mettre des attributs d'inclusion et d'exclusion sur les choix individuels dans une variable de type énumération. Cela peut être utilisé pour, par exemple, n'inclure que des options d'éclairage spécifiques dans certaines familles de décodeurs.

Les Attributs sur un élément <variables> agiront sur tous les éléments <variable> à l'intérieur de celui-ci, ainsi que tout attribut d'inclusion ou d'exclusion sur ces variables individuelles.

Vous pouvez également utiliser le mécanisme d'inclusion pour spécifier la valeur par défaut d'une variable.

Exemple:

      <variable label="My Variable" CV="72" default="3">
        <defaultItem default="12" include="815"/>  
        <decVal/>
      </variable>

Ceci définira la valeur par défaut (initiale) du CV à 12 si le "815" correspond, et utiliser autrement le défaut de 3. Vous pouvez avoir autant d'éléments defaultItem selon les besoins ; le premier qui correspond est utilisé.

Inclure/Exclure pendant la création du volet

La création d'un volet ou d'une partie de volet peut être conditionnée par le type de décodeur en utilisant les attributs "include" et "exclude" sur les éléments<pane> and;group> ..

Un volet ne sera pas créé:

  • Si un attribut d'exclusion est présent, au moins l' ID produit du décodeur ou nom du modèle, ou nom de famille n'est trouvé dans la liste d'exclusion.
  • Si un attribut include est présent, au moins l' ID produit du décodeur ou nom du modèle, ou nom de famille est trouvé dans la liste d'exclusion.
Ces deux conditions doivent être remplies, et il est possible d'utiliser à la fois l'inclusion et l'exclusion sur le même élément, mais c'est plus habituel de n'en utiliser qu'un.

Exemple:

      <pane label="My Pane" include="ModelA,400"> ;
        ...
      </pane>

"Mon volet" ne sera créé que lors de de l'extension de la définition du décodeur pour le modèle A ou l'ID de produit 400.

Vous pouvez aussi créer un élément de type <group> avec include et exclure des attributs. Les éléments à l'intérieur du <group> ne seront créés que si les conditions du <group> sont remplies. Cela peut être utilisé pour, par exemple, afficher des étiquettes de texte spécifiques uniquement pour certains modèles de décodeurs.

Exemple:

      <group include="ModelA,400"> ;
        <label>
          <text>Ces décodeurs sont très vieux.</text>
        </label>
        <label>
          <text>Certaines fonctionnalités sont indisponibles.</text>
        </label>
      </group>

Le texte ne sera créé que lors de l'extension de la définition du décodeur pour le modèle A ou l'ID de produit 400.

Avertissement concernant l'utilisation de l'identification du produit dans Include/Exclude

Il convient de faire preuve de prudence lorsqu'on tente de faire correspondre des attributs d'inclusion ou d'exclusion avec l'ID du produit.. L'ID du produit ne correspond à aucune valeur lue dans le décodeur pendant l'identification mais par rapport à l'attribut productID spécifié dans la définition du décodeur. Cela peut être particulièrement problématique si l'attribut productID a plusieurs valeurs. Par exemple, l'attribut 
  color=;decoder>
    <family name="My Big Family" mfg="NMRA"> ;
      <model model="My Model" productID="400,401" />
      </model>
    </family>
    … ;
  </decoder>
  <pane label="My 400 Pane" include="400"} ;
    … ;
  </pane>
  <pane label="My 401 Pane" include="401"} ;
    … ;
  </pane>
Les deux volets seront toujours créés. Si les définitions doivent différer entre les ID de produits, vous devrez créer deux modèles le premier exemple sous Identification.

Qualification des variables

Variables qualificatives
Un élément "qualifiant" sur un élément de variable vous permet d'activer ou de désactiver la variable en fonction de la valeur actuelle d'une autre variable. Cela peut être utile si, par exemple, le décodeur dispose d'un CV "mode" qui détermine la signification des autres CV.

Si une variable est qualifiée d'inactive, elle n'apparaît dans aucun volet et n'est ni lue ni écrite dans le décodeur et n'est ni lu ni écrit sur le décodeurlorsque vous effectuez des opérations de lecture ou d'écriture de type "Tout" ou "Modifications seulement". La valeur est enregistrée dans le fichier et sera rechargée lorsque l'entrée de l'inventaire est réouverte plus tard.

Dans l'exemple suivant, un CV est interprété comme deux deux variables différentes selon que le décodeur est réglé ou non pour un adressage court ou long.

  <variable label="Variable for Short Address" CV="11">
    <qualifier>
      <variableref>Address Format</variableref>
      <relation>eq</relation>
      <value>0</valeur>
    </qualifier>
    <decVal/>
  </variable>
  <variable label="Variable for Long Address" CV="11" ;
    </qualifier>
      <variableref>Address Format</variableref>
      <relation>eq</relation>
      <value>1</value>
    </qualifier>
    <decVal/>
  </variable>
Les autres opérateurs disponibles sont "gt", "ge", "ne", "lt", "le" et "exists". L'opérateur "exists" vérifie si une variable est définie ou non ; une valeur de "1" signifie qu'elle l'est, et une valeur de "0" qu'elle est signifie que ce n'est pas le cas.

Si plusieurs éléments qualificatifs sont présents, tous doivent être true pour que la variable soit active et affichée.

Volets qualifiants
Un élément "qualifiant" sur un volet vous permet d'activer ou de désactiver le volet en fonction de la valeur actuelle d'une variable.

Si un volet est qualifié d'inactif, son contenu n'est plus affiché. Cela inclut les étiquettes ou autres décorations sur le volet, en plus des variables qu'il contient. En fonction des autres options, le volet peut soit s'afficher comme vide, ou ne pas se montrer du tout. Les variables qu'il contient sont toujours actives, cependant, et un "Lire toutes les feuilles" ou "Ecrire toutes les feuilles". les écrira encore. Si vous avez des définitions de variables qui se chevauchent, examinez attentivement si c'est ce que vous voulez. Vous pourriez avoir besoin de qualifier les différentes variables de chevauchement.

L'exemple suivant permet d'activer l'un des deux volets selon sur le fait que le décodeur est actuellement réglé pour l'adressage court ou long.

  <pane name="Short Only">
    <qualifier>
      <variableref>Address Format</variableref>
      <relation>eq</relation>
      <value>0</value>
    </qualifier>
    <column><row>
      <label label="Short Only Pane"/>
    </row></column>
  </pane>
  <pane name="Long Only">
    <qualifier>
      <variableref>Address Format</variableref>
      <relation>eq</relation>
      <value>1</value>
    </qualifier>
    <column><row>
      <label label="Long Only Pane"/>
    </row></column>
  </pane>
Les autres opérateurs disponibles comprennent "gt", "ge", "ne", "lt" et "le".

Si plusieurs éléments qualificatifs sont présents, tous doivent être true pour que le volet soit actif et affiché.

Elements qualificatifs sur les panneaux
Un élément "qualificatif" sur un élément "label", "soundlabel", "row", "column", "grid" ou "griditem" vous permet d'activer ou de désactiver l'affichage de cet élément sur un panneau spécifique.

L'exemple suivant affiche un avertissement si un élément variable version-number a une valeur trop faible:

    <label label="Numéro de version trop bas !"> ;
      <qualifier>
        <variableref>Numéro de version</variableref>
        <relation>lt</relation>
        <value>3</value>
      </qualifier>
    </label>

Vous pouvez utiliser le même qualificatif pour plusieurs éléments en en incluant les éléments dans un élément "groupe" et en ajoutant le qualificatif de l'élément "groupe":

    <group>
      <qualifier>
        <variableref>Version Number</variableref>
        <relation>lt</relation>
        <value>3</value>
      </qualifier>
      <label label="Version number too low !"/>
      <label label=" "/>
      <label label="Please update."/>
    </group>

Une fois que vous avez lu le numéro de version (variable ou CV) du décodeur, l'étiquette (ou les étiquettes) affichera si cette version est trop faible. Notez qu'initialement, avant que les valeurs soient lue à partir du décodeur, cet affichage dépendra de la valeur par défaut du fichier, ce qui pourrait induire en erreur l'utilisateur user.
